# B70 Hardware Is Functional — The AMG Wall Is a Software Bug

## Summary

After 22 findings building toward the Pioneer-status conclusion that
`aijkokkos + GAMG + SYCL` is not productive on BMG-G31 in May 2026, we
ran three independent SpMV/CG benchmarks **outside the OpenFOAM stack**
to determine whether the AMG `SEGV` of finding 19 is a hardware/driver
issue or a software bug.

**Result:** B70 + oneAPI 2025.3 + SYCL is **fully functional** for
sparse linear-algebra workloads. The AMG wall is a software bug in the
PETSc-Kokkos GAMG construction code, not a hardware limitation.

## Setup

Standalone tests under `/home/heiko/diag/`, kept entirely separate from
the OpenFOAM build to eliminate wrapper-layer effects. Common workload:
1M-row 5-point Poisson stencil (~5M nnz), generated by a 30-line C++
program. Same matrix consumed by all three stacks.

## Results

| Stack | Test | ms / iter | GFLOPS | Effective BW | % of Triad-peak (531 GB/s) |
|---|---|---|---|---|---|
| oneMKL Sparse CG | full CG loop | 0.741 | n/a | 161 GB/s | 30 % |
| **PETSc aijkokkos** | pure SpMV | **0.287** | **35** | **418 GB/s** | **79 %** |
| **Ginkgo dpcpp** | pure SpMV | **0.089** | **112** | **1340 GB/s\*** | peak+ |

\* Ginkgo's effective-BW calculation exceeds Triad-peak because the
1M-element `x` vector (8 MB) fits in B70's L2 cache (12 MB). After the
first read, subsequent SpMV iterations hit cache for `x` while reading
`row_ptr`/`col_idx`/`val`/`y` from VRAM. The reported number is
arithmetic, not physical bandwidth.

## What This Means

- **B70 is not the problem.** All three stacks run, no crashes, no
  driver hangs, no SYCL-runtime instability on the Battlemage hardware.
- **PETSc aijkokkos SpMV reaches 79 % of Triad-peak.** That is healthy —
  comparable to PETSc on a tuned Cuda setup.
- **The GAMG `SEGV` of finding 19 sits in shared PETSc-Kokkos
  initialization code**, not in the SpMV path or in any code that
  depends on hardware behavior. Debug build heals the crash (finding
  20), confirming a memory-safety bug class.
- **The Pioneer status (finding 22) is correct, but for the right
  reason.** The Mai 2026 verdict is about software-stack maturity for
  GAMG on `aijkokkos+SYCL`, not about Battlemage hardware not being
  ready. Eta (finding 21) and Foundation paths run productively.

## Status / Resolution

**Documented.** The hardware/software-bug distinction is now explicit
across all five Mai-2026 finding clusters (15-22 plus 23-26). The
upstream-blocked status of GAMG on `aijkokkos+SYCL` is reaffirmed, with
the Pioneer-frustration redirected to the correct cause: PETSc/Kokkos
GAMG initialization-code memory-safety bug, not Battlemage immaturity.
